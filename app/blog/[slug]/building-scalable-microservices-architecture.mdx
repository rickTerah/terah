# Building Scalable Microservices Architecture: Lessons from Production

As a Head of Engineering, I've had the privilege of architecting and scaling several microservices-based systems. This article shares the practical lessons learned from deploying and maintaining these systems in production environments.

## Why Microservices?

Before diving into implementation details, it's crucial to understand _why_ microservices might be right for your application. At Pariti, we transitioned from a monolithic architecture to microservices to address specific challenges:

- **Team autonomy**: Different teams could deploy independently
- **Technology diversity**: Ability to choose the right tool for each service
- **Scalable deployment**: Scale individual components based on demand
- **Fault isolation**: One service failure doesn't bring down the entire system

However, microservices come with their own complexities that many teams underestimate.

## Core Architectural Principles

### 1. Domain-Driven Design (DDD)

The foundation of any good microservices architecture starts with proper domain modeling:

```typescript
// User Service - bounded context
interface UserDomainService {
  createUser(userData: CreateUserDTO): Promise<User>;
  updateUser(userId: string, updates: UpdateUserDTO): Promise<User>;
  authenticateUser(credentials: CredentialsDTO): Promise<AuthResult>;
}

// Order Service - separate bounded context
interface OrderDomainService {
  createOrder(orderData: CreateOrderDTO): Promise<Order>;
  updateOrderStatus(orderId: string, status: OrderStatus): Promise<Order>;
  getUserOrders(userId: string): Promise<Order[]>;
}
```

Each service should have its own bounded context with clear boundaries. Avoid shared databases between services to maintain autonomy.

### 2. API Gateway Pattern

A robust API gateway serves as your single entry point and handles cross-cutting concerns:

```typescript
class APIGateway {
  constructor(
    private authMiddleware: AuthMiddleware,
    private rateLimiter: RateLimiter,
    private serviceDiscovery: ServiceDiscovery,
  ) {}

  async handleRequest(request: Request): Promise<Response> {
    // Authentication & Authorization
    await this.authMiddleware.authenticate(request);

    // Rate limiting
    await this.rateLimiter.checkLimit(request);

    // Service discovery and routing
    const service = await this.serviceDiscovery.findService(request.path);

    return this.proxyRequest(service, request);
  }
}
```

### 3. Service Communication Patterns

Choose the right communication pattern based on your use case:

#### Synchronous Communication (REST/GraphQL)

```typescript
// Service-to-service HTTP call
const userService = await fetch("https://users-service/api/users/123");
const user = await userService.json();
```

**Pros**: Simple, immediate response
**Cons**: Tight coupling, latency issues

#### Asynchronous Communication (Message Queues)

```typescript
// Event-driven architecture
await messageQueue.publish("user.created", {
  userId: user.id,
  email: user.email,
  timestamp: new Date().toISOString(),
});
```

**Pros**: Loose coupling, resilience, scalability
**Cons**: Complex debugging, eventual consistency

### 3. Circuit Breaker Pattern

Prevent cascading failures with circuit breakers:

```typescript
class CircuitBreaker {
  private state: "CLOSED" | "OPEN" | "HALF_OPEN" = "CLOSED";
  private failureCount = 0;
  private lastFailureTime = 0;

  async execute<T>(operation: () => Promise<T>): Promise<T> {
    if (this.state === "OPEN" && this.shouldRetry()) {
      this.state = "HALF_OPEN";
    }

    if (this.state === "OPEN") {
      throw new Error("Circuit breaker is OPEN");
    }

    try {
      const result = await operation();
      this.onSuccess();
      return result;
    } catch (error) {
      this.onFailure();
      throw error;
    }
  }

  private shouldRetry(): boolean {
    return Date.now() - this.lastFailureTime > this.timeout;
  }
}
```

## Database Strategy

### Database per Service Pattern

Each service should own its own database to ensure loose coupling:

```typescript
// User Service Database Schema
interface UserDatabase {

// Order Service Database Schema
interface OrderDatabase {
  orders: {
    id: string;

    orderId: string;

    price: number;
```

### Data Consistency Patterns

Use the Saga pattern for distributed transactions:

```typescript
// Saga orchestrator for order processing
class OrderSaga {
  async processOrder(orderId: string): Promise<void> {
    try {
      // Step 1: Create order
      const order = await this.orderService.createOrder(orderData);

      // Step 2: Reserve inventory
      await this.inventoryService.reserveItems(order.items);

      // Step 3: Process payment
      const payment = await this.paymentService.processCharge(
        order.totalAmount,
      );

      // Step 4: Confirm order
      await this.orderService.confirmOrder(orderId);

      // Step 5: Release inventory
      await this.inventoryService.confirmReservation(order.items);
    } catch (error) {
      // Compensating transactions
      await this.rollbackOrder(orderId);
      throw error;
    }
  }
}
```

## Common Pitfalls and Solutions

### 1. Distributed Monolith

**Problem**: Services are technically separate but tightly coupled through shared databases or synchronous calls.

**Solution**: Enforce proper bounded contexts and use asynchronous communication where possible.

### 2. Over-engineering

**Problem**: Implementing microservices for a simple CRUD application.

**Solution**: Start with a monolith and split services only when you have clear business boundaries or scaling needs.

### 3. Testing Complexity

**Problem**: Integration tests become complex and slow.

**Solution**: Use contract testing and consumer-driven contracts:

```typescript
// Pact testing example
const provider = new Pact({
  consumer: "orders-service",
  provider: "inventory-service",
  port: 1234,
});

describe("Inventory Service API", () => {
  beforeAll(async () => {
    await provider.setup();
  });

  afterAll(async () => {
    await provider.finalize();
  });

  it("reserves items successfully", async () => {
    await provider.addInteraction({
      state: "items available",
      uponReceiving: "a request to reserve items",
      withRequest: {
        method: "POST",
        path: "/api/reserve",
        body: { items: expectedItems },
      },
      willRespondWith: {
        status: 200,
        body: { success: true, reservationId: "123" },
      },
    });

    const response = await reserveItems(expectedItems);
    expect(response.success).toBe(true);
  });
});
```

## Monitoring and Observability

Implement the three pillars of observability:

### 1. Metrics

Track system performance and business metrics:

- Request rate, error rate, and latency (RED metrics)
- Resource utilization (CPU, memory, network)
- Business KPIs (orders per minute, user signups)

### 2. Logging

Structured logging with correlation IDs:

```typescript
import { v4 as uuidv4 } from "uuid";

// Middleware to add correlation ID
app.use((req, res, next) => {
  req.correlationId = req.headers["x-correlation-id"] || uuidv4();
  res.setHeader("x-correlation-id", req.correlationId);
  next();
});

// Usage in services
logger.info({
  correlationId: req.correlationId,
  userId: user.id,
  action: "order_created",
  orderId: order.id,
});
```

### 3. Tracing

Distributed tracing to follow requests across services:

```typescript
import { trace, SpanKind } from "@opentelemetry/api";

const tracer = trace.getTracer("orders-service");

async function processOrder(orderData: OrderData) {
  const span = tracer.startSpan("process-order", {
    kind: SpanKind.SERVER,
    attributes: {
      "order.value": orderData.totalAmount,
      "order.items_count": orderData.items.length,
    },
  });

  try {
    // Trace service calls
    const inventorySpan = tracer.startSpan("reserve-inventory");
    await inventoryService.reserveItems(orderData.items);
    inventorySpan.end();

    const paymentSpan = tracer.startSpan("process-payment");
    await paymentService.processPayment(orderData.totalAmount);
    paymentSpan.end();
  } catch (error) {
    span.recordException(error);
    throw error;
  } finally {
    span.end();
  }
}
```

## Conclusion

Building and maintaining microservices architecture requires careful planning, robust tooling, and ongoing refinement. The key takeaways from my experience:

1. **Start small** - Don't migrate to microservices just because it's trendy
2. **Invest in tooling** - Observability, CI/CD, and automation are essential
3. **Embrace async** - Use message queues to reduce coupling and improve resilience
4. **Plan for failures** - Assume services will fail and design accordingly
5. **Monitor everything** - You can't improve what you can't measure

The complexity of microservices is justified when you need team autonomy, independent scaling, or technology diversity. For simpler applications, a well-structured monolith might serve you better.

Remember: microservices are a means to an end, not the goal itself. Focus on solving business problems and delivering value to your users.
